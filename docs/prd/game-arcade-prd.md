# 复古游戏角 PRD

## Problem Statement

作为 cutesticky 便签应用的用户，我在学习/工作间隙想用应用里的小游戏放松一下，但目前 `game` tab 只有一个马里奥风格横版跳跃游戏，玩久了会腻。我想要更多经典小游戏（贪吃蛇、俄罗斯方块、打砖块、推箱子、飞机大战等）随手可玩，且能很快切换，作为便签应用里的"复古游戏角"。

## Solution

把 `game` tab 从"直接装一个马里奥"改造成"游戏选择器（卡片墙）"：用户进入 `game` tab 看到一组经典游戏卡片，点击卡片进入对应游戏，游戏内可"返回"回到列表。第一批收录 6 个游戏：贪吃蛇、马里奥、俄罗斯方块、打砖块、推箱子、飞机大战。

每个游戏忠于自己的原版经典美术风格（Nokia 绿屏贪吃蛇、GameBoy 黑白俄罗斯方块、Arkanoid 彩色打砖块、Sokoban 暖色推箱子、1942 蓝紫飞机、马里奥保持现有几何块风），音效用 Web Audio API 合成原版风格的方波/三角波，背景乐用原创短 chiptune 循环（不抄经典旋律，规避版权）。提供全局静音开关。最高分通过 Tauri store 持久化。

## User Stories

### 入口与导航

1. 作为用户，我进入 `game` tab 时看到一张游戏卡片墙，这样我能一眼看到所有可选游戏。
2. 作为用户，我点击某张游戏卡片即进入对应游戏，这样我能快速开始玩。
3. 作为用户，我在游戏内按 `ESC` 或点击"返回"按钮回到游戏卡片墙，这样我能随时切换别的游戏。
4. 作为用户，我从游戏返回列表后再点同一游戏，游戏从头开始，这样不会有残留状态错乱。
5. 作为用户，我切换到其他 tab（如 memo / todo）再切回 `game` tab 时，仍显示游戏卡片墙而非上一局游戏，这样切走即视为离开。

### 游戏内容

6. 作为用户，我能玩贪吃蛇——方向键控制蛇移动、吃食物变长、撞墙或撞自己即结束。
7. 作为用户，我能玩俄罗斯方块——方向键移动/旋转下落方块、整行消除得分、堆到顶即结束。
8. 作为用户，我能玩打砖块——控制底部挡板反弹小球、击碎所有砖块即过关、球掉落即结束。
9. 作为用户，我能玩推箱子——方向键推动箱子到目标点、全部归位即过关。
10. 作为用户，我能玩飞机大战——方向键操控飞机移动、自动/按键射击、躲避并击落敌机、被撞即结束。
11. 作为用户，我能继续玩马里奥——横版跳跃、跳踩敌人、吃金币、到终点旗即过关，行为与现状一致。

### 视觉还原

12. 作为用户，贪吃蛇呈现 Nokia 经典绿屏单色风格，这样我有原版怀旧感。
13. 作为用户，俄罗斯方块呈现 GameBoy 黑白风格，这样我有原版怀旧感。
14. 作为用户，打砖块呈现 Arkanoid 彩色砖块风格，这样我有原版怀旧感。
15. 作为用户，推箱子呈现 Sokoban 暖色调风格，这样我有原版怀旧感。
16. 作为用户，飞机大战呈现 1942 蓝紫弹幕风格，这样我有原版怀旧感。
17. 作为用户，游戏卡片墙的每张卡片展示对应游戏的经典缩略图，这样即便各游戏风格不统一，列表入口仍有视觉秩序。

### 音效与音乐

18. 作为用户，每个游戏都有原版风格的合成音效（吃食物、消行、弹球反弹、推箱、射击爆炸等），这样我有沉浸感。
19. 作为用户，每个游戏有原创短 chiptune 背景循环，这样我玩的时候有氛围。
20. 作为用户，我能通过游戏选择器顶部的全局静音开关一键开关所有声音，这样我在安静环境下不被打扰。
21. 作为用户，默认进入游戏时声音是开启的，这样我能直接体验完整音效。
22. 作为用户，静音开关的状态在同一次会话内对所有游戏生效，这样我不必每个游戏分别静音。

### 最高分与持久化

23. 作为用户，每个游戏记录我的最高分（贪吃蛇长度 / 俄罗斯方块分数 / 打砖块分数 / 推箱子通关步数 / 飞机击坠数 / 马里奥通关用时或金币数）。
24. 作为用户，我的最高分在应用重启后仍然保留，这样我有持续挑战动力。
25. 作为用户，我在游戏卡片或游戏内能看到当前游戏的最高分，这样我知道自己的目标。

### 生命周期与稳定性

26. 作为用户，我从某游戏切走（返回列表 / 切其他 tab / 关窗）后，绝不会再听到该游戏的残留音效。
27. 作为用户，切走游戏后不会因后台仍在跑游戏循环而拖慢应用或耗电。
28. 作为用户，键盘控制只对当前正在玩的游戏生效，不会出现"切走了再按方向键还影响上一局"。

### 可扩展性

29. 作为开发者，我未来想加第七个游戏时，只需在游戏注册清单里加一项 + 新建一个游戏组件，不必改动 `App.vue` 的 tab 结构。
30. 作为开发者，每个新游戏可懒加载，不会拖慢应用首屏。

## Implementation Decisions

### 模块划分

- 新增"游戏选择器"组件 `GameArcade`：渲染卡片墙、管理"当前选中的游戏 / 返回列表"状态、承载全局静音开关。
- 新增 `games` 子目录，承载所有游戏组件与共享模块：
  - 游戏注册清单 `registry`
  - 共享音频工具模块 `audio`（Web Audio 合成音效 + chiptune 背景循环 + AudioContext 生命周期管理）
  - 6 个游戏组件（5 个新写 + 1 个从现有 PlatformGame 迁入改名）
- 改造 `App.vue`：`game` tab 的内容从渲染单一 PlatformGame 改为渲染 `GameArcade`，其余 tab 不动。

### 游戏组件契约（类型形状，源自 grilling 原型）

每个游戏组件遵守统一契约，使 5 个 Agent 可并行实现且可互换：

```ts
// 每个游戏组件
defineComponent({
  props: { muted: boolean },          // 由 GameArcade 传入，控制静音
  // onMounted: 启动游戏循环 + 创建 AudioContext + 绑定键盘
  // onBeforeUnmount（硬规则）: cancelAnimationFrame + 关闭 AudioContext + 移除键盘监听
  // 最高分自存 Tauri store，key: `game:<id>:highscore`
  // 统一快捷键: ESC = 返回列表
})
```

```ts
// 游戏注册清单
type GameEntry = {
  id: string                          // 用于 store key 与路由，如 'snake' / 'tetris'
  name: string                        // 中文显示名
  thumb: Component                    // 卡片缩略图组件（经典风格小预览）
  component: () => Promise<Component> // 懒加载工厂，供 defineAsyncComponent
}
const GAMES: GameEntry[]
```

- 用 `defineAsyncComponent` 懒加载，6 个游戏不一次性打进首屏包。
- 静音开关状态在 `GameArcade` 顶层持有，通过 `props` 下传给当前游戏（或 `provide/inject`），默认开声音。
- 最高分 key 格式 `game:<id>:highscore`，沿用现有 `@tauri-apps/plugin-store` 的 `load('sticky-data.json', { autoSave: true })` 句式，与现有 `memo` / `themeIndex` / `activeTab` 等 key 风格一致。

### 美术与音频决策

- 各游戏忠于原版风格，不强求整体统一；视觉秩序由卡片墙缩略图兜底。
- 音效用 Web Audio API 在代码内合成（方波/三角波/噪声），零资源体积、贴合复古原版。
- 背景乐做原创短 chiptune 循环，**不得照搬受版权保护的经典旋律**（俄罗斯方块的 Korobeiniki、任天堂马里奥 BGM 等）。
- 音频逻辑抽到共享 `audio` 模块，5 个游戏复用同一套合成工具与 AudioContext 管理，避免 5 套重复实现。

### 生命周期决策

- 完全销毁语义：切走/返回/关窗即销毁游戏实例。
- `onBeforeUnmount` 硬规则三件套：`cancelAnimationFrame` + 关闭 AudioContext + 移除键盘监听，缺一不可。
- 关卡进度不持久化（休闲角，重玩成本低）；仅最高分持久化。

### 执行编排决策

- 一次性发布，不分批合并。
- 主流程先搭框架（`GameArcade` + `registry` + `audio` + 契约文档）并把马里奥迁入改名，跑通"选择器 → 马里奥 → 返回"整条链路作为样板。
- 然后 5 个 Agent 并行，各拿契约 + `audio` 模块 + 马里奥样板参照，各写一个新游戏。
- 5 个回来后整合、自测、一次性发布。

### App.vue / 路由影响

- tab 数量不变，仍为 7 个；`activeTab` 的 `'game'` 值语义不变（仍是进入游戏 tab），只是其内容由"单一马里奥"变为"游戏选择器"。
- 现有 `activeTab` 持久化逻辑（切 tab 存 store）不受影响。

## Testing Decisions

### 什么是好测试

只测外部行为与纯逻辑，不测实现细节（不测 Canvas 像素、不测 DOM 结构、不测私有方法）。游戏的可玩性仍以人工试玩为准；自动化测试聚焦"容易回归且可纯函数化"的逻辑核心。

### 测试框架

- 引入 `vitest` 作为单测框架（项目当前零测试基础设施，借此为纯逻辑建立最小测试 seam）。
- 测试 seam 数量目标为 1：所有纯逻辑测试统一在 `vitest` 下运行，`npm test` 一条命令。

### 被测模块与边界

各游戏须把"决策密集"的纯逻辑从渲染/循环中抽离为独立函数，便于单测：

- 贪吃蛇：下一步网格计算、自碰撞判定、吃食物增长。
- 俄罗斯方块：方块旋转矩阵（建议 SRS 或简易旋转）、碰撞判定、整行消除计算。
- 推箱子：推动合法性判定（箱子前方是否可推）、目标点归位判定、关卡完成判定。
- 打砖块：球与挡板/砖块的碰撞反射向量计算。
- 飞机大战：碰撞判定（AABB）、敌机生成调度（纯函数化时间表）。
- 共享 `audio` 模块：工厂返回值形态、静音状态下不创建/不启动播放（行为契约，不测 Web Audio 内部）。
- 共享 `registry`：注册项的 id 唯一性、component 为懒加载工厂。

### Prior Art

项目当前无测试先例（PlatformGame 等均无测试）。本 PRD 引入的 vitest seam 将成为项目首个测试约定，为后续模块（如未来重构 PlatformGame）建立可参照的最小样板。

## Out of Scope

- 各游戏具体还原细节基线（贪吃蛇速度/网格大小、俄罗斯方块旋转系统 SRS vs 简易、打砖块砖块布局与球速、推箱子关卡设计、飞机弹幕模式与难度曲线）——交给对应 Agent 实现时定，本 PRD 仅给方向性建议。
- 每个游戏的具体键位映射（除统一约定 `ESC` 返回外）——交给对应 Agent 定。
- 移动端 / 触屏支持——当前为桌面 Tauri 应用，暂不支持，仅键盘操作。
- 在线排行榜 / 云存档——最高分仅本地持久化。
- 多人对战 / 联机——全部单机。
- 游戏内设置（难度选择、按键自定义、音量分级）——仅全局静音开关。
- 第七个及以后的游戏——架构支持零成本扩展，但不在本次发布范围。
- 为整应用引入端到端测试 / 组件渲染测试——本次只引入覆盖纯逻辑的 vitest 单测 seam。

## Further Notes

- 版权红线：俄罗斯方块 Korobeiniki、任天堂马里奥 BGM 等受版权保护旋律一律不照搬，背景乐必须原创 chiptune。Agent 任务说明须明确此约束。
- 马里奥迁入改名为样板：迁入过程同时作为"游戏组件契约"的可运行参考实现，5 个 Agent 应被引导先读 MarioGame 样板再动手。
- 共享 `audio` 模块的 AudioContext 生命周期是稳定性关键风险点——切走残留音效是最可能的回归 bug，Agent 必须在自测清单中验证"切走后静默"。
- 卡片墙缩略图可由各游戏组件提供"静态预览模式"（非交互的小画面），避免为每张卡片单独维护美术资源。
- 现有 `game-content` 样式类（`App.vue` 中为 PlatformGame 全屏布局而设）需评估是否仍适用于 `GameArcade`，或新增对应布局类。
